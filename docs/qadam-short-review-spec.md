# Спецификация AI-обзора пациента

Статус: `APPROVED — подтверждено командой /build-auto`
Дата: 2026-07-21

## Цель

Добавить в карточку выбранного пациента в кабинете специалиста краткий обзор, который backend формирует на основе переписки, жалоб и ежедневных отчётов. Обзор помогает быстрее войти в контекст, но не заменяет исходные данные и решение специалиста.

## Контракт

- Контракт: `GET /patients/{id}/short-review`, `operationId: shortReview`.
- Роли: `PRACTITIONER` и `ADMIN`; `id` берётся из выбранной записи пациента или введённого администратором UUID.
- Успешный ответ: `ShortPatientReview { statusColor?: GREEN | YELLOW | RED, review?: string }`.
- Запрос проходит через существующий same-origin BFF, access token остаётся в HttpOnly cookie.
- Текст ответа считается недоверенными данными и выводится только как обычный React-текст, без HTML-интерпретации.
- Ошибки авторизации, отсутствия пациента, timeout и временной недоступности преобразуются в локализованные состояния.

## UX

- Размещение: карточка выбранного пациента на `/[locale]/practitioner/patients` и контекст пациента в `/[locale]/admin/dispenses`.
- Заголовок RU: «Краткий обзор пациента»; KK: «Пациенттің қысқаша шолуы».
- Подсказка объясняет источник сводки простыми словами: отчёты, жалобы и переписка.
- Для каждого нового выбранного пациента отображается собственное состояние загрузки.
- У специалиста есть явная кнопка повторного запроса, чтобы получить актуальный обзор.
- Предусмотрены loading, success, empty, timeout и retry/error состояния; сырые `GET`, `POST`, `operationId`, JSON и backend-текст ошибок не показываются.
- Текст обзора сохраняет переносы строк, не выходит за карточку на ширине 360 px и доступен клавиатуре и экранным дикторам.
- Интерфейс не называет сводку диагнозом или клинической рекомендацией.

## Проверка

1. Contract test подтверждает новую OpenAPI-операцию, BFF allowlist и полное покрытие каталога.
2. Playwright с fixture проверяет успешную сводку, повторный запрос, ошибку сервиса, отсутствие технических API-меток и ширину 360 px.
3. Live smoke под синтетическим `PRACTITIONER` подтверждает реальный статус, content type и отображение ответа.
4. `pnpm verify`, practitioner E2E, RU/KK parity, code review и secret scan проходят до push/deploy.

## Критерии готовности

- Live OpenAPI snapshot и generated TypeScript types синхронизированы.
- Операция доступна специалисту из предметной карточки пациента и реально вызывает backend.
- Все состояния локализованы на русском и казахском.
- GitHub `main` и production Vercel содержат проверенный commit.
